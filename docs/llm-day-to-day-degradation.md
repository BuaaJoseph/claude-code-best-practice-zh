# 🔥 Claude Code Ops 4.6 分析。高推理深度

当 Anthropic 发布像 Opus 4.6 这样的模型时，**模型权重**——数十亿学习参数——是冻结的。训练成本极高（数百万美元、数周的计算）。没有人会在一夜之间重新训练模型。

但权重只是更大系统的其中一层。研究揭示了至少 **7 种不同的机制**，即使模型权重冻结，也会导致实际或感知到的质量变化。

| 问题 | 答案 |
|----------|--------|
| 模型权重在发布后会改变吗？ | **否**——所有提供商确认 |
| 模型行为可以日复一日不同吗？ | **是**——已验证 ±8-14% 方差 |
| 是故意的"削弱"吗？ | **否**——无故意退化证据 |
| 基础设施 bug 是真实的吗？ | **是**——Anthropic 确认 3 个 bug 影响高达 16% 请求 |
| 部分是心理因素吗？ | **是**——确认偏差和蜜月效应是真实的 |
| 系统提示词/训练后更改可以改变吗？ | **是**——各提供商有文档记录 |
| 用户应该信任他们的感知吗？ | **部分**——真实原因存在，但感知会放大 |

## 完整的推理栈

模型权重是冻结的，但**其上的九层**可以独立影响你的体验：

```
┌──────────────────────────────────────────────┐
│  你的会话上下文                              │  ← 会话内退化
│  （累积错误、长对话）                         │
├──────────────────────────────────────────────┤
│  系统提示词                                  │  ← 定期更新
│  （安全规则、行为指令）                        │
├──────────────────────────────────────────────┤
│  训练后（RLHF / 微调）                        │  ← 可以静默更新
│  （指令遵循、安全对齐）                        │
├──────────────────────────────────────────────┤
│  采样参数                                    │  ← 可以在服务端调优
│  （temperature、top-p、top-k）               │
├──────────────────────────────────────────────┤
│  投机解码                                    │  ← 草稿模型质量变化
│  （草稿模型预测 + 验证）                       │
├───────────────────────────────────────────���──┤
│  MoE 路由 / 批处理组成                        │  ← 已验证 ±8-14% 方差
│  （每个请求激活哪些专家）                      │
├──────────────────────────────────────────────┤
│  硬件路由                                    │  ← TPU vs GPU vs Trainium
│  （哪个集群服务你的请求）                       │
├──────────────────────────────────────────────┤
│  量化级别                                    │  ← 负载下可能变化
│  （FP16 vs INT8 vs INT4 精度）               │
├──────────────────────────────────────────────┤
│  编译器和运行时                              │  ← XLA bug 已验证真实
│  （XLA:TPU、CUDA、硬件特定代码）              │
├──────────────────────────────────────────────┤
│  模型权重（冻结）                             │  ← 这些不会改变
│  （数十亿学习参数）                            │
└──────────────────────────────────────────────┘
```

关键心智模型：**冻结的权重 ≠ 冻结的行为**。这就像说"相同的发动机 = 相同的驾驶体验"同时忽略了轮胎、道路状况、燃油质量和驾驶疲劳。

## 已验证的原因：基础设施 bug

### Anthropic 的 2025 年 9 月事后分析

2025 年 9 月，Anthropic 发布了详细的事后分析，揭示了**三个独立的基础设施 bug**，这些 bug 在 2025 年 8 月至 9 月间降低了 Claude 的质量。他们的官方声明：

> "我们永远不会因为需求、一天中的时间或服务器负载而降低模型质量。我们用户报告的问题仅由基础设施 bug 造成。"

### Bug #1 —— 上下文窗口路由错误

Sonnet 4 请求被意外路由到配置为 1M token 上下文窗口的服务器，而不是标准服务器。

- **时间线**：8 月 5 日引入，8 月 29 日负载均衡变更后恶化
- **峰值影响**：最严重时 16% 的 Sonnet 4 请求受影响（8 月 31 日）
- **用户影响**：约 30% 的 Claude Code 用户至少收到一条降级消息
- ** insidious 细节**：路由是"粘性的"——一旦命中坏服务器，后续请求继续流向那里
- **修复**：9 月 4-18 日（跨平台推出）

### Bug #2 —— TPU 输出损坏

TPU 服务器上的配置错误导致 token 生成过程中出错，给本应很少出现的 token 分配了高概率。

- **症状**：英文响应中间出现泰语或中文字符，明显的代码语法错误
- **受影响**：Opus 4.1 和 Opus 4（8 月 25-28 日），Sonnet 4（8 月 25 日-9 月 2 日）
- **范围**：仅 Claude API；第三代平台不受影响
- **修复**：9 月 2 日回滚

### Bug #3 —— XLA:TPU 编译器错误编译（最棘手的）

修复精度问题的代码更改意外暴露了 Google XLA:TPU 中的**潜在编译器 bug**。

- **根本原因**：用于选择最可能下一个 token 的近似 top-k 操作"有时返回完全错误的结果，但仅对某些批处理大小和模型配置"
- **为什么难以发现**：它的行为取决于在它之前或之后运行什么操作，以及是否启用了调试工具
- **隐藏数月**：2024 年 12 月的先前变通方法意外掩盖了这个更深的 bug
- **受影响**：Haiku 3.5 已确认；Sonnet 4 和 Opus 3 的子集疑似
- **解决**：从近似 top-k 切换到精确 top-k；接受"轻微效率影响"，因为"模型质量是不可谈判的"

### 为什么发现如此困难

Anthropic 自己的自动化评估没有捕获用户报告的退化，"部分原因是 Claude 通常会从孤立错误中很好地恢复。"每个 bug 在不同平台上以不同速率产生不同症状，创建了"令人困惑的报告混合，没有指向任何单一原因。"

关键背景：Claude 运行在**三个不同的硬件平台**上（AWS Trainium、NVIDIA GPU、Google TPU），每个都有不同的失败模式、编译器和精度行为。你的请求可能在不同日子命中不同的硬件。

## 已验证的原因：MoE 路由方差

现代大型模型通常使用**专家混合（MoE）**架构，其中只有模型参数的子集（"专家"）为每个输入激活。学习路由器决定使用哪些专家。

Scale AI 的研究揭示了关键发现：

> "稀疏 MoE 和批处理推理的组合创建了不可预测的结果，因为批处理的组成可以决定你的查询被路由到哪个专家，而且同一批处理中其他用户查询的混合不是确定性的。"

### 跨提供商测量的日复一日方差

| 提供商 | 日复一日分数方差 |
|----------|--------------------------|
| OpenAI（GPT-4 变体） | ±10–12% |
| Anthropic（Claude 变体） | ±8–11% |
| Google（Gemini 变体） | ±9–14% |

具体示例：同一个模型在一天**越狱抵抗**得分 77%，第二天 63%。同一个模型、相同权重、相同测试——14 个百分点的波动仅来自基础设施。

这意味着即使零 bug 且零变化，同一个模型也可以因为请求如何批处理和路由而在不同日子产生明显不同的质量输出。当日复一日噪音是 10-15% 时，A/B 测试无法可靠地检测到 5% 的质量信号。

## 已验证的原因：系统提示词和训练后更新

### 系统提示词更改

模型权重不改变，但包装这些权重的**系统提示词**可以随时更新。对 Claude 系统提示词演变的分析显示数十次迭代，"热修复"——用于修补不良行为的简短指令——定期添加和删除。

Claude 3.7 的系统提示词包含多个针对常见 LLM"陷阱"的热修复指令。Claude 4.0 的系统提示词删除了所有这些，行为在训练后通过强化学习得到解决。

### 训练后理论

无法解释的质量变化最可能的理论：公司可以在不更改基础模型权重的情况下更新**微调和 RLHF**（来自人类反馈的强化学习）。这在技术上可以说"模型没有改变"，同时仍然通过更新的安全护栏和指令遵循调整来改变行为。

## 已验证的原因：静默模型切换

OpenAI 已被多次记录静默更改用户交互的模型：

- 夜间移除模型选择器，强制用户从 GPT-4o 切换到 GPT-5
- 将 GPT-4o 变成隐藏的"遗留模型"，需要在设置中手动切换，无应用内通知
- "自动切换器"bug 将用户路由到错误的模型
- Plus 订阅用户报告模型未经同意切换到"受限版本"

Sam Altman 承认推出"比我们希望的更坎坷"。Reddit 帖子的数千个赞成票称新模型为"灾难"和"降级"。

这表明模型切换**确实在行业中发生**——有时是故意的（产品决策），有时是意外的（路由 bug）。

## 贡献因素

### 负载下的量化

为了经济高效地为数百万用户提供服务，公司可能会服务**量化**的模型——将精度从 FP16 降至 INT8 或 INT4。这可以将内存使用减少 2-4 倍并加速推理，但会引入微妙的质量损失。提供商是否在负载下动态切换量化级别存在争议，但技术能力存在，并在 vLLM 和 TensorRT 等服务框架中有充分文档记录。

### 投机解码

现代服务栈使用较小的"草稿"模型来预测多个 token，然后让真正的模型验证它们。理论上这保留了相同的输出分布，但实践中接受率因域和上下文而异。开箱即用的草稿模型在某些情况下可能工作良好，但通常在特定领域任务或非常长的上下文中表现不佳。

### 上下文窗口污染

在长编码会话中，早期的错误在上下文中累积。模型看到自己的错误并可能延续。这是在单个会话中最常见的"Claude 变笨了"的原因——不是模型退化，而是上下文污染。

**实用技巧**：当质量感觉不对时，使用 `/compact` 或开始新会话。这是你可以做的最可操作的事情。

## 斯坦福研究 —— 以及为什么它很复杂

2023 年斯坦福大学和加州大学伯克利分校（Chen、Zaharia、Zou）的里程碑研究——"ChatGPT 的行为是如何随时间变化的？"——经常被引用为 LLM 退化的证据。标题发现：

> GPT-4 在"这个数是质数吗？逐步思考"上的准确率从 **97.6% 降至 2.4%**，在 2023 年 3 月至 6 月之间。

### 研究证明了什么

- "相同"LLM 服务的行为**可以在短期内大幅改变**
- 不同能力可能朝相反方向移动（GPT-4 数学变差，GPT-3.5 变好）
- 代码生成质量下降（GPT-4 可执行代码：52% → 10%）
- 研究创造了术语 **"LLM 漂移"**

### 方法论批评

- 3 月版本使用 **temperature 0.0**，而 6 月版本使用 **temperature 1.0**——增加随机性的基本混淆变量
- 每个任务仅 **500 个查询**——对于确定性统计声明来说太小
- "数学问题"实际上是是/否问题，其中模型的猜测模式改变了，而非其数学能力
- 变化可能反映了故意的**训练后安全更新**，而非退化

研究证明了重要的事情——LLM 行为随时间改变——但机制可能是故意的更新，而非无意退化。

## 心理因素

### 确认偏差

一旦有人发推"Claude 今天很笨"，你开始注意到每一个错误。在没有人抱怨的日子里，你忽略同样的错误。社交媒体放大了这个效应。

### 蜜月效应

用户在与新模型的蜜月期间体验初始期，然后逐渐发现限制。模型没有改变——期望��能力增长得更快。

### 任务难度方差

你的任务日复一日不同。困难问题的一天感觉模型变差了，即使它没有。

### "周末 Claude"神话

尽管许多用户相信按日期的模式，但严格分析**没有找到按日期质量模式的一致证据**。一项名为"AI 周一更笨"的分析一无所获。

### LLM 的随机性质

LLM 是随机的。同一个提示每次可以产生不同的输出。在运气不好的时候，你可能会连续获得几个差的响应——纯随机，而非退化。

## 底线

用户描述的现象**真实但归因错误**：

- **正确**：他们的经验在某些日子确实退化了
- **不正确**：模型被故意"削弱"

真正原因是以下组合：

1. **基础设施 bug** —— Anthropic 事后分析已验证（高达 16% 请求受影响）
2. **MoE 路由方差** —— Scale AI 测量 ±8-14% 质量波动，即使零变化
3. **系统提示词和训练后更新** —— 各提供商有文档记录
4. **硬件异质性** —— TPU vs GPU vs Trainium，每个都有不同的失败模式
5. **上下文污染** —— 长会话内会话质量下降
6. **确认偏差** —— 社交媒体放大了感知模式
7. **随机方差** —— 同一个模型、同一提示，每次输出不同

测量问题很严重：±8-14% 的日复一日方差意味着你无法将真实的 5% 质量变化与噪音区分开来。这就是为什么"全在你的脑海中"和"他们削弱了它"两派都信心十足——信噪比使得从个人经验中无法判断。

# 🔥 Codex 5.3 高推理和发现

### 报告范围

本节解释为什么用户可能在短期内经历 Claude 输出质量下降，而 Codex 5.3 在编码任务上感觉稳定或更��。��点不在永久模型质量排名。关注短期生产行为在真实服务条件下的表现。

报告日期：2026 年 3 月 5 日。

### 观察到的模式

报告的模式是：

1. 一段时间内质量可接受。
2. 质量似乎下降了几天。
3. 质量接近之前的基准水平恢复。

这种形状通常是服务栈或推出模式，而非永久基础模型能力变化。永久能力下降通常不会在没有明确回滚或修复的情况下这么快恢复。

### 高推理：为什么 Codex 5.3 在糟糕的窗口中看起来更好

Codex 5.3 在另一个提供商的退化期间看起来明显更强，可能出于以下技术原因，可以同时发生：

1. **产品-目标契合**。Codex 5.3 针对代码生成和代理编码工作流进行了优化，因此即使原始模型强度相同，由于工具编排、仓库推理和以代码为中心的指令调整，也可以产生更好的编码结果。
2. **推理策略差异**。提供商独立调优延迟、推理深度和解码默认值。一个提供商更保守的策略对于同一天来说可能看起来比另一个激进的speed优化策略"更聪明"。
3. **服务路径分离**。即使两个提供商托管最先进模型，它们运行不同的路由层、编译/运行时栈和推出管道。一个栈中的事件并不意味着另一个相关退化。
4. **推出和回滚时机**。如果一个提供商正在推出而另一个稳定，用户可能在没有底层模型权重长期变化的情况下看到大的临时质量差异。
5. **会话级污染效应**。在长编码聊天中，错误累积可以放大感知下降。一个失败的竞争助手可能感觉更好，仅仅因为失败的会话被重置或其工具循环恢复得更快。

### 详细发现

对于"Claude 感觉非常弱了大约四天，然后恢复了"这样的报告，最可能的解释是：

1. 提供商端事件、路由问题、解码/运行时 bug 或推出回归影响了请求子集。
2. 问题持续足够长到在工作流中反复被注意到。
3. 问题被修复或回滚。
4. 感知质量迅速恢复。

在同一期间，Codex 5.3 可能感觉明显更好，因为它没有走相同的事件路径，而且编码任务优化放大了实际结果中的差距。

### 此模式的假设排序

| 假设 | 可能性 | 理由 |
|------------|------------|-----------|
| 提供商事件加回滚 | 高 | 最符合多天下跌后快速恢复的模式 |
| 服务配置更改（采样/延迟/推理预算） | 高 | 常见的行为突然变化来源，无需模型再训练 |
| 静默别名或快照移动 | 中高 | 可以在没有可见用户操作的情况下改变行为 |
| 仅提示漂移和上下文污染 | 中 | 可能使会话退化，但不太可能单独解释广泛的多天报告 |
| 永久基础模型退化 | 低 | 与快速恢复到之前质量不一致 |

### 什么会确认或证伪这一发现

要将这从高置信度推断转化为硬证明，收集跨天的相同任务集的请求级遥测：

1. 请求时的确切模型标识和快照/别名。
2. 提供商暴露的任何后端指纹或发布标记。
3. 解码参数（temperature、top_p、top_k、max tokens）。
4. 延迟、超时和错误率追踪。
5. 固定编码基准提示集上的结构化质量分数。
6. 失败点的会话长度和 token 上下文深度。

如果质量下降与事件窗口、配置更改或后端指纹变化相关，则事件/配置假设得到确认。如果不存在这种变化且退化仅在长会话中，则上下文污染成为主要解释。

### 实际工程指导

为了减少生产中的日复一日方差：

1. 当可用时固定模型快照而非使用浮动别名。
2. 存储请求元数据（模型 ID、参数、延迟、错误、响应质量标签）。
3. 运行固定的每日编码任务金丝雀套件并在回归时警报。
4. 几个失败回合后重置或压缩长时间运行的会话。
5. 为事件窗口保留备用提供商/模型路径。
6. 在内部仪表板中将"模型质量"与"服务可靠性"分开。

### 最终结论

Codex 5.3 在短期 Claude 退化窗口中看起来更好是现代 LLM 操作中技术上可信且预期的结果。最强的解释不是永久模型崩溃。最强的解释是一个提供商临时服务路径退化，结合同一期间另一个提供商编码特定优化和稳定运营。
